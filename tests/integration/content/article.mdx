<p className="scribe-lead">BitTorrent’s peer protocol sounds more violent than it is. A peer can be <code>choked</code>, <code>interested</code>, or both—and none of those words describe the TCP connection itself.</p>

The vocabulary describes **permission and demand**. Once those axes are separated, the state machine becomes small enough to hold in your head and strict enough to encode directly in Rust.

> A connection tells us who is reachable. Peer state tells us whether either side currently has a reason—and permission—to exchange pieces.

## Two independent questions

Every established peer connection carries four booleans. They answer two questions in both directions:

1. **May data move?**
   - The remote peer may choke or unchoke us.
   - We make the same allocation decision for the remote peer.
2. **Would data be useful?**
   - We become interested when the peer owns a missing piece.
   - They become interested when we advertise something useful.

<Callout variant="insight" title="Choking is allocation, not punishment">
  A peer can keep a healthy connection open while refusing piece requests. That lets it preserve protocol context while reallocating upload capacity every few seconds.
</Callout>

## Encoding the transition

The wire decoder should return a transition, not mutate the scheduler directly. That keeps message framing deterministic and makes allocation policy independently testable.

```rust filename="crates/styx-wire/src/peer/session_state_transitions.rs" lineNumbers highlight="4-10" focus="3-12" remove="6" add="8-9"
pub enum PeerEvent {
    Choke,
    Unchoke,
    Interested,
    NotInterested,
    Have { piece: u32 },
    Bitfield { bytes: Vec<u8> },
    Request { piece: u32, offset: u32, length: u32 },
    Cancel { piece: u32, offset: u32, length: u32 },
    Piece { piece: u32, offset: u32, block: Vec<u8> },
}

let pending_block_requests_by_remote_peer_and_piece_index = requests.partition_point(|request| request.piece <= announced_piece);
```

<Callout variant="warning" title="Frames may outlive assumptions">
  A peer can disconnect between reading the length prefix and reading the payload. Never hold a scheduler lock across socket I/O, and never assume a partial frame can be resumed on a new connection.
</Callout>
